# Bobob Design System

- Visual changes must protect the current product loop: the homepage should make the web-operations workflow obvious first, with URL/header/DNS/sitemap/token checks exposed as compact task paths; Blog + Play pages should still show readable content, Play/result/share actions, and related links before decorative or support content; tool pages must preserve input, output, copy/review actions, and next-tool paths.
- Blog + Play game screens should feel like compact web experiments and old web mini-games, not landing pages or quizzes: one clear rule, one primary direct interaction, stable canvas dimensions, immediate mouse/touch/keyboard feedback for arcade entries, mobile fallback controls, and no donation/ad/support prompts in the first pass. Arcade entries should generally be `1분` score/time runs or clearly endless `계속` loops.
- Do not use one shared game skin for every Play entry. Keep the outer Bobob shell quiet and consistent, but choose each game surface from its `planningBrief.initialVisualDirection`, then deepen it with web search, comparable screenshots, and the actual mechanic. Lottery/chance can start from ledgers, tickets, odds, and recovery slips; growth from workshops, upgrade rows, meters, and bottlenecks; hunting/action from field HUDs, target distance, danger lanes, and hit feedback; sorting/puzzle/memory/management from desks, boards, consoles, or control sheets. These are starting metaphors, not locked components.
- Favor simpler and rougher game surfaces over busy polished dashboards: one primary surface, one compact numeric/status strip, restrained palette, readable controls, and genre-specific object shapes. Avoid generic card grids, repeated result panels, glow-heavy badges, and visual decorations that make different games feel identical.
- Home Featured Play should render the current Play registry in the compact card grid instead of capping at the original five MVP games; keep the grid dense and neutral when the list grows.
- After changing CSS, layout, theme, or background behavior, verify the page in a real browser on `localhost`; if a build ran, restart the local server first to avoid stale CSS/JS chunk regressions.
- Use shadcn/ui-style local primitives from `apps/main/src/components/ui`.
- Keep the product-lab tone: neutral zinc palette, compact panels, crisp borders, modest radius, no raised colored top borders, and no decorative accent strips.
- Dark and light themes need visible workbench layers: use surface-shell, raised/sunken surface, strong neutral border, panel ring, and subtle shadow tokens so sidebars, center panels, primary work areas, and support sections do not visually merge.
- Scrollbars should be thin, quiet, rounded, transparent-track, and token-driven across WebKit and Firefox. Avoid bulky native-looking scrollbar rails, but keep hover contrast visible enough for long JSON/code/tool outputs.
- Avoid oversized heroes, gradient-heavy sections, emoji tool cards, bento marketing layouts, nested cards, explanatory UI copy, and AI-generated-looking card accents.
- Decorative WebGL and pointer-reactive backgrounds are disabled because they caused scroll/input lag. Do not reintroduce ReactBits, Light Rays, Galaxy, `ogl`, shader canvases, `data-reactbits-background`, or pointermove/requestAnimationFrame background listeners without explicit approval.
- Do not add `framer-motion` or another animation dependency for decorative backgrounds without explicit approval.
- Background treatment should stay static and token-based: use `var(--background)`, surface tokens, borders, shadows, and scrollbars for depth instead of animated canvas or pointer-following layers.
- Background edits must keep `npm run harness:pointer-background` green; the harness verifies removed WebGL/pointer background assets stay absent.
- Tool pages use the same frame: left navigation, top context bar, central tool panel, right examples/FAQ/guides/related tools.
- Tool detail center panels are task-first: render the working tool surface immediately after the detail header, then place compact quick-start examples, use cases, localized failure cases, pre-copy checklist items, and related next actions below it. Input/output should never be buried under support material.
- The primary input/output surface should be framed as one distinct work area with stable `data-primary-work-area` and `data-primary-work-content` hooks. Keep related tools, examples, review checks, and mobile reference accordions in a separate `data-tool-support-sections` group so the page does not read as one continuous mixed list.
- Quick-start and review sections should stay list-like, neutral, and separated with simple neutral boundaries instead of nested cards or decorative strips.
- Core tool result summaries should be compact and task-specific before any raw JSON block. Regex result details should use neutral rows for snippets, matches, positions, and capture groups rather than a large decorative panel; high-use regex snippets and the regex-from-examples generator should stay inside Regex Tester as compact review panels, not separate thin pages. JSON Formatter summaries should show examples, root type, depth, top-level shape, sensitive-looking key, empty value, duplicate key, and large array diagnostics, a copyable API response report with safe sharing checks, JSON path preview with copyable paths, an inline path inspector, copy actions for formatted/minified/extracted values, and parse error line/column context with localized repair hints before the formatted output. JWT Decoder summaries should expose token status, algorithm, iat/nbf/exp/lifetime time-window summary, expected issuer/audience/scope comparison, Authorization header/token/payload/redacted payload copy actions, a copyable JWT auth report, registered/public/private claim inspection, sensitive claim detection, identity claims, and review warnings before raw JSON tabs. Base64 summaries should show variant, byte count, padding, normalized decode, URL-safe output, decoded content shape, JSON key count, formatted JSON preview, JWT segment signal, image/data URL preview, image download, copyable data URL output for detected images, a copyable Base64 payload report with safe sharing checks, secret-like keyword warning, control-character count, and decoded JSON/binary/image warnings in compact neutral rows before the copy block. Cron summaries should show preset buttons, five-field breakdown, selectable day-of-month/day-of-week OR/AND semantics, browser timezone runtime context, selectable preview timezone, next run estimates, a copyable Cron schedule report with safe deployment checks, and warnings as compact neutral panels before schedule copying. UUID summaries should show count presets, format, uniqueness, version, variant, and bulk warnings. Hash summaries should show example inputs, digest/HMAC mode selection, HMAC secret diagnostics, input metrics, per-algorithm notes, a copyable Hash signature report with safe sharing checks, and security warnings. Password summaries should show random-password and passphrase modes, character groups, ambiguous-character exclusion, passphrase word count/separator/number controls, entropy, charset or word-list size, local-generation status, a copyable Password safety report with safe secret exclusion, and storage/length/passphrase compatibility warnings. Random Token summaries should show session/API/CSRF/webhook presets, byte length, format, entropy, encoded length, URL-safe status, padding, intended use, local-generation status, a copyable Random token report with safe sharing checks, and storage/format warnings. QR Code summaries should show URL/Wi-Fi/email/vCard examples, payload builder fields, error correction, image size and quiet-zone controls, payload type, destination host, tracking parameter count, scan density, a copyable QR scan report with safe sharing checks, PNG actions, and scan/privacy warnings. Color summaries should show examples, swatches, HEX/RGB/HSL support, WCAG contrast status, AA/AAA diagnostics, alpha limitations, luminance gap, contrast warnings, and a copyable Color accessibility report. DNS summaries should show examples, status cards, record-type diagnostics, deployment checklist for A/AAAA/CNAME/NS/TXT/DMARC, readable record rows, server-route/public-host warnings, and raw JSON last. HTTP summaries should show status, redirect chain, hop timing/cache details, redirect diagnostics, allowlisted final response headers, pasted-header parser, security header readiness, and CSP generator as neutral sections before raw JSON. Meta Tag and Open Graph summaries should show title and description length, canonical/page host, image host, robots policy, HTTPS/local-private host checks, noindex risk, image-format warnings, and copyable crawler reports before raw tags. robots.txt and sitemap generator summaries should show crawl-readiness diagnostics for URL validity, directive counts, generated entries, duplicate removal, mixed hosts, non-HTTPS entries, and local/private hosts before raw output. SQL/CSS/JavaScript formatter summaries should show realistic examples, input/output metrics, known limitation warnings, and risky pattern warnings before formatted output. SQL formatter surfaces should include query type, table-reference chips, JOIN/subquery/parameter counts, WHERE/LIMIT clause checks, sensitive-field signals, schema-change warnings, and mutation-without-WHERE warnings. CSS formatter and minifier surfaces should include selector preview, custom property count, color token count, at-rule count, compression ratio, duplicate selector warning, ID specificity warning, and comment-removal warning. JavaScript formatter and minifier surfaces should include import/export count, async/await count, browser API signals, fetch warning, console warning, compression ratio, TODO warning, eval warning, and comment-removal warning.
- SQL formatter report surfaces must include a copyable SQL review report with query shape, table references, clause checks, review notes, and a safe execution checklist.
- CSS formatter report surfaces must include a copyable CSS review report with selector, token, cascade, specificity, compression, review notes, and a visual review checklist.
- JavaScript formatter report surfaces must include a copyable JavaScript review report with module/runtime signals, browser API/fetch/eval risks, review notes, and a safe review checklist.
- JSONPath summaries should stay inside JSONPath Tester as compact neutral rows with examples, supported syntax chips, match metrics, matched paths and values, copy matched paths, no-match warnings, and truncation warnings before the copy-ready result.
- URL Parser summaries should stay task-first: examples, URL structure metrics, query parameter rows, tracking detection, clean URL copy, canonical candidate, HTTPS/local/private-host/credential/hash warnings, and a copyable URL canonical report should appear before raw JSON.
- Tool directory pages at `/tools` and `/{locale}/tools` use the dense product-lab card system, and the home page may expose selected operations-first workflows as the primary service while Blog/Play remain supporting content sections.
- Tool directory acquisition clusters should be compact list-like panels that reuse localized tool copy and related next-action links; do not turn them into marketing cards.
- workflow recipe cards should stay compact, local-first, and list-like. They should connect registered tools into task paths for API response formatting, token debugging, redirect debugging, DNS deployment checks, Wi-Fi QR creation, secure token generation, security-header review, deploy config validation, and CSV cleanup for API payloads without becoming marketing cards or standalone recipe pages.
- task-shaped search queries may surface workflow recipes above tool results, but the recipe UI should use neutral borders, short step chips, and localized step reasons rather than large hero cards.
- Search result match-signal chips should stay compact and list-like, using localized result context from titles, descriptions, use cases, examples, failure cases, pre-copy checklist items, aliases, intents, keywords, FAQ, guide titles, and related next-action context without becoming decorative tag clouds.
- Related next-action links can appear in search results and directly below a tool panel, but keep them compact and list-like rather than another nested card surface. They should show a short localized use-case line, not only the related tool name.
- Desktop tool pages require real resizable left and right sidebars with localStorage persistence: left 280px default / 220px min, right 340px default / 280px min, center 560px target min. The right reference panel is closed by default and opens with a localized top-bar toggle; when closed, the center work area reclaims the width.
- Resizable sidebars must clamp to the workbench container width. Narrow desktop widths must not create horizontal overflow; let the center panel shrink before the shell breaks.
- Desktop tool pages must use one aligned workbench shell around the resizable panels. Do not wrap left, center, and right panels in separate outer rounded bordered cards.
- Resize handles use a transparent hit area with one stable neutral divider line; avoid hover backgrounds, divider color transitions, duplicate center-panel inset side lines, delayed layout restore, or extra border columns that visually protrude or flash when users click between tool pages.
- Detail top bars must include a locale-aware brand/home link so users can return to the main screen without opening side navigation.
- Tool navigation clicks must preserve sidebar scroll position across tool detail navigation. Use non-scrolling tool links and restore the navigation scroll container from localStorage.
- Recent, favorite, and tool session sections should be compact utility UI and must use localStorage only; do not add analytics, account state, or server persistence for repeat-usage UX. Favorite save/remove actions and tool session restore-last-work/clear-local-history actions should use icon buttons with localized visible text where space allows.
- Tool detail share actions such as copying the current tool link should stay compact in the top action group, use lucide icons, localized labels, a stable QA hook, Clipboard API plus fallback, and browser-only behavior.
- JSON Escape, YAML Validator, and ENV Parser Validator working surfaces should match the core detail pattern: compact example buttons, metric tiles, localized warnings, structured preview lists, then copy-ready raw output. YAML Validator should include Docker Compose Validator/Formatter checks in the same surface with compact service rows and formatted YAML output instead of a separate thin Compose page.
- CSV JSON Converter should include the CSV Cleaner workflow in the same surface with compact options, metrics, preview rows, and localized warnings rather than creating another thin CSV page.
- Markdown Previewer should include the Markdown Table Generator workflow in the same surface with compact examples, delimiter selection, row/column metrics, warnings, copy-ready table output, and preview insertion rather than creating another thin Markdown page.
- Demand tier is not a user-facing label. Do not show demand wording or raw `core` / `growth` / `long-tail` badges in cards, search results, or detail headers.
- `contentCluster` is not a user-facing badge. Use localized category labels in cards and tool detail sections instead of raw cluster slugs.
- Mobile must keep Sheet navigation and single-column content instead of draggable panels.
- Mobile tool detail pages are task-first: show the primary tool surface before support/reference material, collapse examples, failure cases, pre-copy checks, FAQ, guides, and related tools into a compact mobile reference accordion, and use a sticky mobile action bar only for real navigation/state actions.
- Desktop tool detail pages must make the primary input/output surface visually dominant. Keep examples, failure cases, pre-copy checks, recipes, guides, and related tools in a support accordion or right reference panel so they do not visually attach to the working surface.
- common result cards and diagnostic cards should frame copy-ready output, metric summaries, and warning lists with neutral borders, clear headings, and stable QA attributes. Do not let output blocks look like another support/example card.
- Do not surface revenue-operation language in UI: no AdSense/review-status/approval/monetization wording, fake ad placeholders, or demand-tier labels in public screens.
- Real ad units may reserve space only with a valid publisher id and slot id. Keep allowed placements separated from controls: after the primary tool result area, at the bottom of the right reference panel, or in long guide content. Disabled or unconfigured ad units must render nothing visible, and page-level auto ads must not be initialized by the app shell.
- Use lucide icons for actions where an icon exists.
- Light/Dark/System is required product behavior. Use next-themes, ThemeProvider, ThemeToggle, and class-based `.dark` variables.
- Theme changes must be verified with `npm run harness:theme`.
- Layout changes must be verified with `npm run harness:layout`.
- Visual layout changes must also run `npm run harness:visual` against a running app for desktop, mobile, and RTL screenshots.
- If UI policy changes, update AGENTS.md and this skill in the same change.
