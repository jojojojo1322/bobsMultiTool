## 서버를 붙이고 싶어지는 순간

뭔가 웹서비스를 만든다고 하면 서버부터 떠오른다. 로그인, 저장, 랭킹, 댓글, 결제, 관리자 화면 같은 것들이 줄줄이 따라온다. 이런 기능은 언젠가 필요할 수 있다.

문제는 첫 단계에서 붙이면 사이트의 중심이 바뀐다는 것이다. 글을 쓰고 Play를 만드는 일보다 계정 상태, 데이터 저장, 예외 처리, 약관 범위가 먼저 커진다. 그러면 “오늘 하나 더 올리기”가 어려워진다.

## 지금은 정적으로 충분한 것

bobob.app의 1차 목표에는 서버가 꼭 필요하지 않다.

- Blog는 MDX 파일로 충분하다
- Play는 JSON 데이터와 재사용 엔진으로 충분하다
- 결과는 브라우저 상태로 보여주면 된다
- 공유는 링크와 텍스트 복사로 충분하다
- 유입 확인은 sitemap, feed, Search Console로 시작하면 된다

여기까지는 정적 배포가 오히려 낫다. 빌드 결과가 예측 가능하고, 배포가 단순하고, 서버 비용이나 장애도 줄어든다.

## 서버가 필요한 때

나중에 랭킹이나 사용자 저장이 진짜 의미를 만들 때는 서버를 붙여야 한다. 댓글도 마찬가지다. 결제나 AI 생성처럼 API key를 숨겨야 하는 기능도 서버가 필요하다.

하지만 아직은 아니다. 지금은 우선순위를 나누는 편이 더 중요하다. 그래서 오늘 일감 세 칸 보드 같은 Play도 이 방향과 맞다. “막힌 일, 시간을 잡을 일, 오늘 판 밖으로 뺄 일”을 나누는 감각이 개발에도 그대로 필요했다.
